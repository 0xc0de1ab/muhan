# `internal/world`

이 패키지는 게임 공간을 묘사하는 도메인 모델, 런타임 메모리 상태(State), 그리고 초기 로딩 로직을 포함합니다.
전통적인 MUD 게임의 '갓 객체(God Object)' 문제를 해소하기 위해 도메인 별 세부 락(Fine-grained Lock)을 구현했습니다.

## 핵심 구조

- **`model/`**
  - 데이터의 순수 스키마 및 상수들.
  - `PlayerID`, `RoomID`, `CreatureID` 등의 식별자 타입 체계와 `ClassAssassin` 등의 직업 상수가 중앙 집중식으로 관리됩니다.

- **`state/` (메모리 락과 런타임 상태)**
  - 가장 중요한 부분으로, 메모리상의 실시간 게임 데이터를 들고 있습니다.
  - 기존에는 전체 World를 통째로 잠그는(Global Mutex) 방식이었으나 성능 이슈를 줄이기 위해 분할되었습니다.
    - `player.go`: 접속자 및 캐릭터 데이터 관리 (`playersMu`)
    - `room.go`: 방 데이터 및 내부 오브젝트/시체 관리 (`roomsMu`)
    - `creature.go`: 몬스터 등 NPC 데이터 관리 (`creaturesMu`)
    - `combat.go`: 전투 상태 및 관련 타이머 관리 (`combatMu`)
    - `board.go`: 게임 내 게시판 및 메시지 메모리 관리 (`boardsMu`)
  - **Copy-on-Write 패턴**: `Player(...)`와 같은 조회 메서드 호출 시 포인터를 그대로 반환하여 경합 상태가 발생하는 것을 방지하기 위해 데이터를 복사해서 반환하는 안전한 패턴이 적용되어 있습니다.

- **`load/`**
  - 서버 구동 시 레거시 C 파일 시스템(바이너리 파일들) 혹은 JSON 파일들로부터 데이터를 읽어들여 객체화하는 초기화 모듈입니다.

## 상태 관리(State)의 목적

- C 기반의 오래된 아키텍처에서는 포인터를 서로 난잡하게 참조했지만, Go 환경에서는 Data Race 발생 소지가 높습니다.
- `state` 패키지는 엄격한 Mutex 잠금 안에서만 데이터를 읽고 쓰도록 강제하여 `Data Race`를 원천 차단하고 서버 다운 위험성을 낮춥니다.
